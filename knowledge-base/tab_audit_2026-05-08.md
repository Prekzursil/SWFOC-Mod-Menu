## Why this doc exists

`editor-100.md` acceptance criterion #3 requires:

> All 22 tabs have a UX audit pass complete: each tab is reviewed for cohesion,
> redundancy, label clarity, and grouping; consolidation decisions are documented
> in `knowledge-base/tab_audit_2026-05-08.md` (NEW). Acceptable outputs are:
> KEEP, MERGE-INTO-X, SPLIT-INTO-X+Y, RETIRE.

The audit *substance* already exists: `tests/SwfocTrainer.UiTests/WpfTabAuditTests.cs`
drives every tab of `MainWindowV2` via FlaUI/UIAutomation and emits a full
per-tab interactive-control inventory to `knowledge-base/wpf_ui_audit_2026-04-27.md`
(22 tabs, 1872 controls captured). What was missing was the **spec-named decision
document** — a per-tab KEEP/MERGE/SPLIT/RETIRE verdict synthesized from that
inventory. This doc is that artifact. It records *decisions*; it does not execute
them. Consolidation execution is the separate iter-290 / iter-291 work in the spec
iter chain.

## Method

- **Source 1 — runtime control inventory:** `wpf_ui_audit_2026-04-27.md`, the
  auto-generated FlaUI walk. Provides per-tab control counts, GroupBox section
  names, and bounding boxes — the objective grouping signal.
- **Source 2 — XAML structure:** `src/SwfocTrainer.App/V2/MainWindowV2.xaml`
  `<TabItem Header=...>` declarations (confirms the current live tab set).
- **Verdict basis:** for each tab, four lenses — *cohesion* (do the groups belong
  together?), *redundancy* (does another tab already do this?), *label clarity*
  (is the header + group naming unambiguous?), *grouping* (are the GroupBox
  sections sensible?).

## Verdict rationale — the four non-KEEP decisions

### #5 Probes & Scripts → MERGE-INTO Lua Playground

The clearest redundancy in the trainer. Both tabs are a Lua REPL: select/type a
Lua expression, run it, see output, save it as a named recipe.

- **Probes & Scripts:** Canned recipe picker → Lua input box → Output (last 40)
  → Save-as-recipe.
- **Lua Playground:** Recipe library → Lua input → Last response → "Iter 100-300
  LIVE wires" preset panel.

Lua Playground is the strict superset (it adds the LIVE-wire discovery presets).
Merge Probes & Scripts' "Canned recipe" + "Recent" history into Lua Playground's
"Recipe library", retire the duplicate input/output panel. Net: one Lua surface
instead of two near-identical ones.

### #15 Battle Control → MERGE-INTO Combat

Battle Control (58 controls: `Target`, `One-click`, `Unit cap override`) is
entirely combat-domain. `One-click` is a small set of convenience macros;
`Unit cap override` is a single setting. Combat is the dedicated, mature combat
tab (135 controls, 5 cohesive groups). Fold Battle Control's three groups in as
a "Battle macros" GroupBox under Combat. No functional overlap is lost; one
fewer combat-themed tab to choose between.

### #16 Story Events → MERGE-INTO World State

Story Events (42 controls — the smallest baseline tab: `Fire event`,
`Set story flag`) duplicates World State's existing **"Story events and maphack"**
GroupBox, which already carries a `Fire event` control. World State is also the
home of the related "Cinematic & Audio Controls" and "Engine: Trigger Victory"
groups. Move `Set story flag` into World State's story group; Story Events
becomes redundant.

### #21 Cross-Faction → MERGE-INTO Unit Control (conditional)

Cross-Faction (43 controls: `Source unit (must be local)` → `Recruit
destination`) recruits a unit across factions — i.e. it changes a unit's
allegiance. Unit Control already exposes **"Change unit owner →"**. The two are
the same operation surfaced twice.

**Conditional:** Unit Control is already the largest tab (190 controls). This
merge should land *after* the recommended relocation of Unit Control's off-domain
"Spawn unit" group to the Spawning tab (see cross-cutting findings below) so the
net control count stays manageable. If that relocation is not done first,
prefer leaving Cross-Faction as a standalone tab.

## Cross-cutting redundancy findings (de-dup, not tab removal)

These are duplicate **GroupBoxes across KEEP tabs** — they do not change a tab
verdict but should be cleaned up during iter-290/291 consolidation:

| Duplicated surface | Appears in | Recommendation |
|---|---|---|
| Credits / Tech level write | Player State (`Credits`, `Tech level`) **and** Economy (`Credits + Tech`) | Economy is the complete economy surface (scope + multipliers + global). Keep economy writes in Economy; Player State should keep only slot-identity, diplomacy, and the hero/respawn convenience. |
| Diplomacy (`Make ally/enemy`) | Player State · World State (`Diplomacy`) · Galactic (`Diplomacy`) | Three copies. Designate World State `Diplomacy` as canonical; have Player State / Galactic deep-link to it or drop their copies. |
| Time scale / game speed | Speed (`Global game speed`) · Director Mode (`Time scale`) · Connection & Diagnostics (`Time scale` probe button) | Speed is the dedicated tab. Remove Director Mode's `Time scale` group; the Diagnostics button is a read-only probe and may stay. |
| Hide/Show GUI | Connection & Diagnostics (`Hide GUI`/`Show GUI`) · Director Mode (`Hide UI`) | Remove Director Mode's `Hide UI` group; keep the Diagnostics buttons. |
| Spawn unit | Unit Control (`Spawn unit` group) · Spawning (whole tab) | Relocate Unit Control's `Spawn unit` group into Spawning. This also unblocks the conditional Cross-Faction merge (#21). |
| Hero respawn | Player State (`Heroes` group) · Hero Lab (`Hero actions`) | Hero Lab is the dedicated hero tab. Player State `Heroes` may stay as a 2-control convenience or deep-link to Hero Lab. |

Director Mode's verdict is **KEEP** rather than MERGE precisely because, once its
`Time scale` and `Hide UI` duplicate groups are stripped, its remaining
`Add waypoint` + `Playback` surface is a genuinely distinct cinematic-timeline
workflow with no peer tab.

## Landing-page decision (pre-answers iter-298)

The spec iter chain (iter-298) asks whether a dedicated **Home** tab improves
cohesion. The spec directs that decision be documented here.

**Decision: NO dedicated Home tab.** Rationale:

1. **Connection & Diagnostics** is already the correct default landing tab — an
   operator must verify the bridge connection before anything else works, and
   that tab surfaces pipe-reachable status, version/build probes, and quick
   links to the capability and surface reports.
2. **Quick Actions** already *is* the ≤2-click operator-workflow surface the
   spec mission asks for — each button is a composite of 2-7 primitive bridge
   calls for a common operator goal (god mode, drain enemies, reveal galaxy…).
3. Adding a Home tab would make it a 25th tab — net negative against the spec's
   explicit "reduce UI clutter / fewer tabs" goal. The two tabs above cover the
   landing role between them.
